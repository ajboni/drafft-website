# From Tiles to Edges: Getting Lost Building a Level Editor

I started making a game.

Which is already suspicious. It’s been ages since I last made a game. I’ve been pretty busy working on Drafft.  But I had an idea, some momentum, and the dangerous thought:

> “I just need a simple level editor.”

That sentence should come with a warning label.

---

## Stage 1 — Denial (Plain Text Everything)

The first version was as naive as it gets.

I described rooms using plain text:

```
..###..
..#.#..
..###..
```

- `.` = floor
- `#` = wall

It felt great. Fast, visual, no tooling required. However, the charm quickly faded as I realized this method lacked essential features: metadata for rooms, structured data management, and scalability.

So I did what everyone does next.

---

## Stage 2 — Bargaining (Excel)

I moved everything into a spreadsheet. Each cell now represented a tile, allowing me to introduce structure and coordinates into my designs

- `.` floor
- `#` wall
- maybe `$01` for some special marker

Now I had structure, metadata, and a clear grid. I could easily annotate rooms, define coordinates, and even add properties to specific tiles. It felt like I had finally found the right approach.

And it _worked_… for a bit.

Then the cracks started showing.

### The Wall Problem

Walls were tiles.

Which means this:

```
. # .
```

The wall **occupies a full tile**.

That sounds fine until you think about what a wall actually is.

In a dungeon crawler (or even a hotel corridor):

- walls are **boundaries**, not spaces
- you don’t _stand_ on a wall tile
- you stand **next to it**

So now I had:

- fake geometry
- weird spacing
- rooms that didn’t feel right

And worse…

### The "Thick Walls" Problem

At some point I started questioning the walls themselves.

If walls are tiles, they’re _thick_.

Which means:

- doors live inside tiles
- walking through a door means stepping into a “wall cell”

That felt wrong.

In something like a corridor:

- you walk **through** a doorway
- you don’t step _into_ a wall tile and then out again

I tried a few mental gymnastics to justify it:

- special door tiles
- invisible transitions
- pretending the wall tile isn’t really there

None of it felt right.

It always came back to the same issue:

> I was modeling space incorrectly.

### The Gap Problem

If I remove wall tiles:

```
(2,2) floor
(2,3) wall
(2,4) floor
```

If I drop the wall tile:

```
(2,2) floor
(2,4) floor
```

Now rendering at `y * tileSize` creates a **gap**.

So the grid is no longer truthful.

Which means I need a _derived grid_.

Which means I’ve already lost control of my own data.

---

## Stage 3 — Realization (Walls Are Not Tiles)

The turning point came with an unexpected realization: walls aren't tiles; they are edges. This fundamental shift meant rethinking my entire approach. Instead of thinking in terms of filled spaces, I began to consider the boundaries between them.

```
[#]
```

You think like this:

```
[cell] | [cell]
```

The wall lives **between cells**.

---

## Stage 4 — Acceptance (Edges, Not Tiles)

So I rebuilt the model.

Cells became:

```ts
Cell = {
  label?: string
}
```

Edges became:

```ts
Edge = {
  type: "wall" | "door" | "window",
};
```

And the key idea:

```ts
edges["6,6:S"] = { type: "door" };
```

Meaning:

- cell (6,6) south edge has a door
- that same edge is shared with the cell below

A single source of truth for boundaries.

---

This model matches how spaces actually behave:

- Cells = spaces you can occupy
- Edges = boundaries between spaces

This model allowed for a more intuitive representation of spaces and their connections, making corridors and rooms logical and cohesive

---

## The Hidden Win: Data Starts Making Sense

This is where things got interesting.

With tiles, everything was visual-first.

With edges + cells, it became **data-first**.

Now I can ask questions like:

- “Can I move from A to B?”
- “Is there a door here?”
- “What defines this room?”

And answer them cleanly.

Example:

```ts
function canMove(x, y, dir) {
  const edge = getEdge(x, y, dir);
  return edge?.type !== "wall";
}
```

Clean logic, fewer special cases.

---

## Stage 5 — Scope Creep (The Editor Appears)

At this point I had:

- spreadsheets
- parsing logic
- edge reconstruction

And I thought:

> “I should just build a simple editor.”

That was a mistake.

---

## Stage 6 — Acceptance 2.0 (It’s an Editor Now)

Inside my own app.

With:

- a grid
- cell annotations
- edge painting (walls, doors, etc)

What started as:

> “just enough to test maps”

Turned into:

- a custom data format
- rendering logic
- interaction systems

The game?

Still waiting.

---

## What I Learned (The Hard Way)

### 1. Your data model matters more than your rendering

If your data is wrong, everything downstream becomes a workaround.

This is something I’ve been thinking a lot about while building Drafft. Not just for maps but for any game-related data such as items or dialogue.

### 2. Tiles are intuitive, but misleading

They’re great for visuals.
They’re bad at representing relationships.

### 3. Edges unlock clarity

The moment you treat boundaries as first-class data:

- things simplify
- logic becomes predictable
- systems compose better

### 4. Tools will emerge whether you want them or not

If your workflow is painful, you _will_ build tooling.

The only question is whether you admit it early or after 3 rewrites.

---

## Where I Ended Up

II started with the intention of creating a game, not an editor. Yet, this path revealed a deeper truth: how we structure our data shapes our entire approach to game design. What began as a spreadsheet-like representation has now transformed into a world I can explore and expand upon—a true testament to the power of thoughtful data modeling.

---

## Closing Thought

In retrospect, my initial suspicion about making a game wasn't unfounded. However, it led me to an editor that not only serves its purpose but also enriches my understanding of game development's intricate dance between logic and creativity.
