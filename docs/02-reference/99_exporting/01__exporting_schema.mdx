# Output Schema

This document describes the structure of the output schema for Drafft, including scripts, dialogues, actors, quests, items, grids, kanban, and miscellaneous entities.

> Work in progress!

## Schema Export

Beyond the exported data payload itself, Drafft can generate and export a JSON Schema (Draft 2020-12) representation of the current payload:

- Automatic sidecar mode: enable **Include JSON schema sidecar on export** in Project Settings > Export Options. Every game export will also write a sibling `.schema.json` file.
- On-demand mode: click **Export Schema Now** in the same Export Options section to generate only the schema and choose where to save it.

The schema is generated from the current exported data and selected sections. Collections with dedicated schema modules (Maps, Variables, Scenarios) produce typed, rule-driven schemas. Collections without a dedicated module fall back to the common `BaseDocument` shape.

## Collections Overview

The application consists of the following primary collections:

- **Scripts**: Contains script files and folders for code components
- **Dialogues**: Contains dialogue trees and conversation structures
- **Actors**: Contains character definitions and their properties
- **Quests**: Contains quest definitions and condition/property data
- **Items**: Contains item definitions and custom data payloads
- **Variables**: Contains author-defined state defaults
- **Scenarios**: Contains scenario diffs used for simulation state setup
- **Scenes**: Contains playlist-style chains of scripts and dialogues
- **Canvas**: Contains canvas documents for visual note/workflow authoring
- **Maps**: Contains tile/edge map data
- **Misc**: Contains miscellaneous design data documents
- **Grids**: Contains spreadsheet-like data structures
- **Kanban**: Contains kanban board task management structures

## Common Properties

All objects in every collection share these common properties:

| Property      | Type           | Description                                                      |
| ------------- | -------------- | ---------------------------------------------------------------- |
| `name`        | string         | Name of the object                                               |
| `type`        | string         | Type of the object (Script, Folder, Dialogue, Actor, Grid, etc.) |
| `parent`      | string \| null | ID of the parent object, or null if no parent                    |
| `icon`        | string         | Icon identifier for the UI representation                        |
| `color`       | string         | Color code in hex format                                         |
| `path`        | string         | Path to the object in the tree structure                         |
| `isClonable`  | boolean        | Whether the object can be cloned                                 |
| `isRemovable` | boolean        | Whether the object can be removed                                |
| `isReadOnly`  | boolean        | Whether the object is read-only                                  |
| `collection`  | string         | The collection this object belongs to                            |
| `modified`    | string         | Timestamp of last modification                                   |
| `_id`         | string         | Unique identifier for the object                                 |

:::note
`locked`, `lockedBy`, and `_rev` are stripped from all exported documents and will not appear in the export payload.
:::

## Variables Collection

Variables are exported as parsed JSON content and represent default authored state. Each Variable document carries a `_sid` field (normalized state identifier) in its content, which acts as a unique key for state resolution.

When schema export is enabled, each Variable document gets its own typed `$defs` entry (e.g. `Variable_player`, `Variable_world`) with property types inferred recursively from the authored default values. The `variables` array in the schema uses a `oneOf` discriminated union keyed on `content._sid`.

## Scenarios Collection

Scenarios are exported as parsed JSON diffs. They are intended to represent overrides relative to authored defaults.

## Scenes Collection

Scenes export as ordered playlists of script/dialogue references.

## Canvas Collection

Canvas documents export as node/edge content payloads.

## Maps Collection

Maps export as cell and edge data. Internal editing-only metadata is stripped from map cells/edges during export.

## Misc Collection

Misc documents are exported with parsed content when supported by editor language.

### Column Object

Each column holds an array of cards.

| Property | Type     | Description              |
| -------- | -------- | ------------------------ |
| `id`     | `string` | Unique column identifier |
| `title`  | `string` | Title of the column      |
| `type`   | `string` | Always `column`          |
| `cards`  | `array`  | List of card objects     |

### Card Object

Each card tracks a task or idea.

| Property      | Type      | Description                        |
| ------------- | --------- | ---------------------------------- |
| `id`          | `string`  | Unique card identifier             |
| `type`        | `string`  | Always `card`                      |
| `title`       | `string`  | Title of the card                  |
| `description` | `string`  | Full description in Markdown       |
| `tags`        | `array`   | List of string tags (e.g., "Idea") |
| `related`     | `array`   | List of related entity IDs         |
| `completed`   | `boolean` | Whether the task is completed      |
| `dueDate`     | `string`  | Optional due date (YYYY-MM-DD)     |
| `assigned`    | `string`  | Optional assignee name             |
